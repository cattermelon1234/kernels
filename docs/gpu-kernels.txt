some key kernel optimization techniques: 

- shared memory (L1 cache), tiling

loading from global memory is slow. Every memory load from global memory takes hundreds of cycles. 
This is not solely a GPU problem, but a general computer architecture bottleneck. DRAM is 
much farther away from the compute units than the registers and caches. Additionally, DRAM is designed
for high throughput rather than low latency. Combined with possible cache misses, page faults, and TLB 
misses, accessing global memory is very expensive, and is often the bottleneck of many GPU kernels. 

In CPU performance engineering, we typically try to mitigate this by optimizing for cache locality, 
prefetching, and minimizing cache misses. Modern GPUs give us a special type of memory called "shared memory", 
which is actually a partition of the L1 cache given explicitly to the programmer to control. As a result, 
loading data from shared memory is much faster than loading from global memory: think of it as accessing data 
from the L1 cache instead of from DRAM. The key to optimizing GPU kernels is to minimize global memory accesses and maximize shared memory usage. 

For every kernel, we have to eventually load every single element from the global memory into the registers to 
perform computations. The key win in shared memory is that oftentimes, kernels reuse the same data multiple times. 
Take the GEMM (general matrix multiply) kernel for example. In naive gemm, thread (i, j) computes the output scalar 
at C[i][j] = A[i][:] * B[:][j]. But notice that thread (i, j+1) also needs to compute C[i][j+1] = A[i][:] * B[:][j+1]. 
Similarly, every single thread in the same row and column will load the same row of A and same column of B from global 
memory, leading to a huge number of redundant global memory accesses. 

We would love to load A into shared memory and B into shared memory once, and work from there. Unfortunately, the size of 
shared memory is very limited. The bigger shared memory gets, the fewer blocks we can have resident on the SM at the same time,
leading to low occupancy and underutilization of the GPU. Thus, we would like to only load small tiles of A and B into 
shared memory at a time. This is what "tiling" is. We break down the output matrix C into small tiles, and have each thread block 
compute one tile of the output matrix C.

- coalesced memory access 

A SM runs warps of 32 threads concurrently. When threads in a warp access global memory, the GPU will try to coalesce those memory accesses into as few transactions as possible.
If the threads in a warp access contiguous memory addresses, then the GPU can coalesce those accesses into a single transaction, which is much faster than having each thread 
access memory independently. However, if the threads in a warp access non-contiguous memory addresses, then the GPU will have to issue multiple memory transactions, which 
is bad for performance. Thus, we want to ensure that threads in a warp access contiguous memory. Often, this means accessing memory in a 
row-major order in memory, which is the default layout for C/C++ arrays.

This is also the reason why we try to use striped access patterns when loading data into shared memory. If each thread were instead to load 
a contiguous chunk of data, the coalescer is unable to detect the contiguous access pattern, and we end up with multiple transactions instead of one. 
A properly coalesced access pattern looks like: 
load A[threadIdx.x * 32 + i] 
thread 0: loads A[0], A[32], ...
thread 1: loads A[1], A[33], ...
thread 2: loads A[2], A[34], ...
....
thread 31: loads A[31], A[63], ...

The coalescer is able to coalesce into: load A[0:31], A[32:63], ... which is much more efficient.

- kernel fusion 

Kernel fusion combines multiple kernels into a single kernel. Again, inference and many GPU workloads are 
memory-bound, not compute-bound. Every time we run a kernel, we have to pay an overhead cost, load our 
tensor data from global memory into registers, and write the results back to global memory. 
If we can combine multiple kernels into a single kernel, we can reduce the number of times we have to load 
and store data from global memory, which can significantly improve performance.

Kernel fusion is not always helpful. Reductions like sum, max, min, softmax, etc. are good examples of when 
not to fuse kernels. These operations reduce many input elements into a smaller output, which requires synchronization 
and communication across threads (or even thread blocks). Fusing additional work before or after the reduction often increases 
register pressure, complicates synchronization, and can reduce parallelism. Instead, it is usually better to implement the reduction 
as its own highly optimized kernel and then launch a separate kernel for the subsequent elementwise operations. Modern GPU libraries often fuse only the inexpensive elementwise operations surrounding a reduction (such as bias addition and activation), while keeping the reduction itself as a standalone kernel.

some good candidates for kernel fusion: 
bias + activation 
gemm/linear + bias + activation
quantization + gemm + dequantization

- warp-wide reduction 
warp-wide reduction is a GPU intrinsic. An intrinsic is a function that is implemented directly in hardware and is usually much faster 
than a software implementation. It works by halving the number of elements in a warp with each iteration in parallel until we have 
a single element, as shown below. 

ex: 
__device__ __forceinline__
float warp_reduce_sum(float value) {
    for (int offset = 16; offset > 0; offset >>= 1) {
        value += __shfl_down_sync(0xffffffff, value, offset);
    }
    return value;
}

- BLOCK WIDE REDUCTION 

block-wide reduction is reducing elements from a block of GPU threads to a single element. Examples include summations, min/max, 
vector norms, etc. Usually, each block in a GPU kernel is responsible for a contiguous tile of data, and each thread is responsible for 
a couple elements. If we want to find the sum of all elements in a tensor, we would usually reduce by block, then reduce further. 
Block-wide reductions are usually implemented using shared memory. There are two main ways to do efficient block-wide reductions.

1. With tree-based reduction, we can reduce the number of elements by half in each iteration. Each thread loads its element into shared memory, 
then we iteratively reduce the number of elements by half until we are left with a single element. This is usually done in a loop, 
where each thread reduces its element with the element of another thread. 

2. With warp-based reduction, we can reduce the number of elements by half in each iteration, but we do not need to use shared memory.
Instead, we can use warp shuffle intrinsics to reduce the number of elements within a warp efficiently and store the result in shared memory. 
Then, we can use another warp reduction to reduce these results into a single element. This is usually the best way to do block-wide reduction 
since usually, a block has a small number of threads (256 or 512 commonly), which is 4 or 8 warps. This method only fails if we have 
more than 32 warps in a block, which is rare in practice.

- vectorized loads
A vectorized load is when we load multiple elements from memory into a register with a single instruction.
The key to a vectorized load is a "trick", which is we can interpret a contiguous chunk of memory as a 
vector of a certain type. For example, if we have an array of floats (4 bytes each), we can instead 
interpret this array as an array of float4 (16 bytes each). Then, we can load a float4 from memory into a 
register with a single instruction, effectively loading 4 floats at once.

This reduces the number of memory transactions and can significantly improve performance. Each thread can now load 
4 elements from global memory with a single instruction, and the coalescer can coalesce these vectorized loads into even fewer transactions.

int tid = blockIdx.x * blockDim.x + threadIdx.x;

// Treat the float array as an array of float4s
float4* A_vec = reinterpret_cast<float4*>(A);

// Each thread processes 4 floats
float4 v = A_vec[tid];

// Update all 4 elements in registers
v.x += 1.0f;
v.y += 1.0f;
v.z += 1.0f;
v.w += 1.0f;

// Write back with a single vectorized store
A_vec[tid] = v;

- loop unrolling 
Loop unrolling is simply where we manually "unroll" loops by writing the for loop out manually for 
a fixed number of iterations. This can improve performance because it reduces the overhead of loop 
control (incrementing the loop variable, checking the loop condition, and branching back to the top of the loop).
Branching can be expensive on the GPU due to the BTB and predictor failing. 

#pragma unroll 
for (int i = 0; i < TILE_SIZE; i++) {
    // compute products using data in shared memory
}

- pipelining

GPU pipelining works much like CPU pipelining. In CPU pipelining, we have multiple stages of instruction execution so that 
while one instruction is being executed, another can be decoded, and another can be fetched. This allows us to keep every stage 
of the CPU busy, improving performance. 

In GPU pipelining, we have multiple stages of computation in a kernel. For example, in a GEMM kernel, we have the following stages:
1. load tile of A and B into shared memory
2. compute products using the data in shared memory
3. write results back to global memory 

These stages can be pipelined, so while a tile is being computed, another is being loaded into shared memory, and another is 
being written back to global memory. This allows us to keep the GPU busy and improve performance. 

before: 
1 2 3 
      1 2 3 
            1 2 3

after:
1 2 3 1 2 3
X 1 2 3 1 2
X X 1 2 3 1

- async memory copy, double buffering

Async memory copying and double buffering are techniques that utilize GPU pipelining. 

We already know that global memory access is expensive. If a GPU kernel is memory-bound, that means that most of the warps are sitting 
idle, waiting for memory accesses to complete. Async memory copying and double buffering are techniques to mitigate this. 
The idea is that while we are computing the results using one tile of data in shared memory, we can asynchronously begin loading the next 
tile of data into shared memory, as explained in pipelining. This way, we can overlap computation with memory access, keeping the GPU busy and improving performance. 

To do this, we use two buffers. The first buffer is used for computation, while the second buffer is used for loading the next tile of data. 
CUDA provides async memory copy functions, such as cudaMemcpyAsync, allowing us to overlap host-device transfers with kernel execution.
Modern GPUs also provide cp.async (exposed through cuda::memcpy_async), which allows data to be copied directly from global memory into
shared memory asynchronously within a kernel.

With double buffering, while one shared-memory buffer is being used by tensor cores or CUDA cores for computation, the other buffer is
simultaneously being populated with the next tile of data from global memory. Once computation on the current tile is complete, the roles
of the two buffers are swapped. The previously loaded buffer becomes the compute buffer, and the old compute buffer becomes the load buffer.

Without double buffering, execution typically follows a pattern of:
```
    Load Tile 0
    Compute Tile 0
    Load Tile 1
    Compute Tile 1
    ...
```
With double buffering, execution can be pipelined as follows:
```
    Load Tile 0
    Compute Tile 0, Load Tile 1
    Compute Tile 1, Load Tile 2
    Compute Tile 2, Load Tile 3
    ...
```

In the actual implementation, this pipeline follows a production-consumption model.
```
producer_acquire(next_buffer); // producer (memory loader) acquires next buffer for loading
async_copy(tile_next, next_buffer); // producer starts load process
producer_commit(next_buffer); // producer finishes issuing async copies

consumer_compute(cur_buffer); // consumer does calculation on cur buffer, already ready from last iter
consumer_release(cur_buffer); // consumer signals it is done, ready for buffer to be reused for next iter load

consumer_wait(next_buffer); // consumer waits for producer to finish loading next buffer before computing
```

- shared bank conflicts, padding/swizzling
GPU shared memory is divided into multiple banks. Each bank can service one request per clock cycle. Thus, multiple banks 
can be accessed in parallel in the same clock cycle. However, if multiple threads in a block access the same shared 
memory bank, a bank conflict occurs and accesses are serialized -> bad performance. 
Recall how in a CPU, only one thread can access a cache line at a time. Similarly, in a GPU, only one thread can access a shared memory bank at a time.

GPUs typically have 32 banks of shared memory, 1 for each thread in a warp. GPUs typically have 32 shared memory banks, one for each thread in a warp. 
Shared memory addresses are assigned to banks in a round-robin fashion:

bank = (address / sizeof(element)) % 32

The size of the element matters because we wouldn't want half of a float to be in one bank and the other half to be in another.
This also means consecutive elements are placed in consecutive banks. 

Index:  0  1  2  3  ... 31 32 33 ...
Bank:   0  1  2  3  ... 31  0  1 ...

Most naturally written cuda avoids bank conflicts. Each thread accesses shared[threadIdx.x], which naturally leads to each thread in a warp 
accessing a different bank. However, issues can arise with different strided access patterns. 

shared[threadIdx.x * 32] leads to:
Thread: 0 1 2 3 ...
Index : 0 32 64 96 ...
Bank : 0  0  0  0 ...

This is the worst case situation. Every thread in the warp accesses the same bank, this is terrible performance. You might be thinking: 
who would access memory in this strange pattern? This most commonly appears in operations like transpose for example.
In transpose kernels, we often have to access shared memory in a column-major order, which leads to multiple threads in the 
same warp accessing the same bank.  

Padding and swizzling are two techniques to avoid bank conflicts. Padding is simply adding extra elements to the shared memory array. 
Instead of shared[32][32], we can use shared[32][33]. This way, when we access shared[threadIdx.x][i], we are accessing different banks for each thread in the warp.
This simple trick avoids bank conflicts and improves performance, and it is also very low effort :)

Swizzling is a more complicated technique that involves rearranging the data in shared memory to avoid bank conflicts.
While padding is usually sufficient, swizzling can be used in more complicated access patterns.

- tensor core utilization 
Tensor cores are specialized matrix multiplication units introduced in NVIDIA's Volta architecture. They are basically just a specialized 
ALU unit that SPECIFICALLY does gemm. Tensor cores are much faster than CUDA cores for matrix multiplication, but they have some limitations.
Tensor cores can only operate on certain data types (FP16, BF16, INT8, etc.) and require the matrices to be in a specific layout (row-major or column-major).

We can utilize tensor cores by using the WMMA (Warp Matrix Multiply and Accumulate) API, which allows us to perform matrix multiplication on small tiles of matrices using tensor cores.
ex: 
or example, a common WMMA operation is

    C(16×16) = A(16×16) × B(16×16) + C(16×16)

where all 32 threads in the warp participate in computing the 16×16 output tile.

Example:

```cpp
using namespace nvcuda;

wmma::fragment<wmma::matrix_a, 16, 16, 16, half, wmma::row_major> a_frag;
wmma::fragment<wmma::matrix_b, 16, 16, 16, half, wmma::row_major> b_frag;
wmma::fragment<wmma::accumulator, 16, 16, 16, float> c_frag;

wmma::fill_fragment(c_frag, 0.0f);

wmma::load_matrix_sync(a_frag, A, lda);
wmma::load_matrix_sync(b_frag, B, ldb);

wmma::mma_sync(c_frag, a_frag, b_frag, c_frag);

wmma::store_matrix_sync(C, c_frag, ldc, wmma::mem_row_major);

- TMA (Tensor Memory Accelerator)

The Tensor Memory Accelerator (TMA) is a hardware unit introduced in NVIDIA's Hopper architecture (H100, SM90) that efficiently copies multidimensional tensors from global memory to shared memory.
TMA is primarily used in high-performance tiled kernels such as GEMM, FlashAttention, convolutions, and other tensor operations that repeatedly stage large tiles into shared memory.

TMA is basically just the successor to cp.async from the ampere architecture. 

- cp.async (Ampere): Threads in a warp cooperate to issue asynchronous copy instructions.
- TMA (Hopper): A single thread launches the transfer, and dedicated hardware performs the copy.

With cp.async, every thread in a warp spends instructions issuing memory copies. With TMA, only one thread launches the transfer while a dedicated copy engine moves the entire tensor tile. This reduces instruction overhead, frees the warp to perform computation, and allows memory transfers to better overlap with execution.

Unlike a normal memcpy, TMA understands multidimensional tensors. Instead of copying a flat array of bytes, it can copy an entire matrix or tensor tile using information such as its dimensions, strides, and memory layout.

Typical data flow:

Global Memory -> Tensor Memory Accelerator -> Shared Memory
